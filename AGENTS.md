# app-best-llm - 项目指令

本项目是一个 AI 工具智能推荐应用，通过联网调研为用户推荐最合适的 AI 工具组合。

## 项目目标

构建一个基于实时联网搜索的 AI 工具评测与推荐系统。

核心价值：**禁止依赖预设知识库**，所有推荐必须基于实时搜索的权威评测数据、社区反馈与官方文档。

## 核心工作流

当用户提出 AI 工具推荐需求时，按以下四步流程执行：

### 需求画像

通过 3-5 个问题建立完整用户画像，分两个层次收集信息：

**基础维度：**
- 任务类型：代码开发 / 文档写作 / 学术研究 / 创意设计 / 深度调研 / 多媒体创作 / 数据分析
- 技术水平：新手 / 中级 / 专家
- 预算范围：免费 / 低成本(<¥100/月) / 专业级(¥100-500/月) / 企业级(>¥500/月)
- 使用频率：偶尔 / 经常 / 重度

**深度维度（不可省略，是推荐质量的核心依据）：**
- **当前痛点**：现有工具/方法哪里让你不满？（例："上下文窗口太小" / "响应太慢" / "代码补全不准"）
- **已有工具栈**：已在用哪些 AI 工具？需要与哪些系统集成（IDE/Slack/Notion 等）？
- **决策优先级**：在性能 / 价格 / 易用性 / 数据隐私 / 生态完整性中，排序最重要的 1-2 项

### 候选工具发现

**广度搜索（识别 ≥5 个候选工具）：**
- `[任务类型] AI 工具推荐 2026` 和 `best AI tools for [任务类型] 2026`
- `[用户痛点关键词] AI solution`（针对痛点定向搜索）

**排除策略：**
- 与用户已有工具栈高度重叠的工具降低优先级，主动搜索差异化替代品

### 深度调研

对每个候选工具执行以下四个维度调研：

**性能评测（必须获取具体数值）：**
- 权威 benchmark：SWE-bench / Chatbot Arena / LiveCodeBench / MMLU（根据任务类型选相关项）
- 要求：必须记录具体数值（如"SWE-bench 45.3%"），**禁止使用"性能强大"类空洞描述**
- 无 benchmark 数据时：改用 GitHub Stars 趋势 + 社区活跃度替代，并明确说明

**竞品差异对比：**
- 搜索 `[工具A] vs [工具B] [任务类型]` 获取第三方横向比较
- 明确记录每个工具的**独特优势**和**已知短板**

**隐藏成本挖掘：**
- API 配额限制、速率限制、超额费用（官方文档确认）
- 数据隐私政策（企业/敏感数据场景尤为重要）
- 迁移成本：学习曲线、与用户现有工具栈的集成难度

**生态健康度：**
- 维护活跃度：GitHub 最近更新频率 / 官方博客更新频率
- 社区规模：Discord / Reddit / GitHub Discussions 活跃用户数
- 企业背书：母公司稳定性、近期融资状况

**社区口碑（重点搜索与用户痛点相关的评价）：**
- Reddit（r/MachineLearning / r/artificial / 领域专属子版块）、Hacker News、知乎
- 优先寻找具体使用场景下的真实反馈，而非泛泛好评

**交叉验证：** 关键数据点需至少 2 个独立来源验证，冲突时标注争议点。

**异常处理：**
- 搜索无结果 → 标注"信息不足"，建议用户提供更多上下文
- 无 benchmark 数据 → 改用替代指标，并明确说明
- 价格无法确认 → 标注"价格信息不可用，建议直接访问官网"

### 方案输出

以结构化 Markdown 格式输出推荐报告并保存为文件，包含：
- **查询日期**（所有时效性数据必须标注）
- **用户场景摘要**（任务类型 / 技术水平 / 预算 / 决策优先级）
- **首选方案 + 备选方案 1-2**：工具名、benchmark 数据（具体数值）、推荐理由、隐藏成本提示、月费用
- **横向对比矩阵**：关键维度对比各工具
- **"为什么没选"**：针对用户场景说明未推荐工具的具体原因
- **调研来源**：benchmark 数据链接、社区反馈来源、价格信息来源

**输出前质量门控（自检清单）：**
- [ ] 推荐理由引用了具体数据点（数值 benchmark 或具体用户评价），无空洞描述
- [ ] 横向对比矩阵已填写，各工具的差异清晰可辨
- [ ] "为什么没选"部分针对用户场景，而非通用描述
- [ ] 隐藏成本和避坑提示已包含（尤其关注用户痛点对应的风险）
- [ ] 工具组合建议与用户已有工具栈兼容

## 关键约束

- **禁止依赖预设知识**：所有数据必须来自本次搜索
- **时效性标注**：价格、功能等信息必须标注查询日期
- **评测来源质量**：权威 benchmark > 知名技术媒体 > 社区讨论，排除未验证来源
- **具体化原则**：禁止"性能强大" / "功能丰富"类空洞描述，必须给出具体数据或案例

## 默认设置

- **语言**：始终使用简体中文与用户对话（用户明确要求其他语言时除外）
- **搜索策略**：优先本地上下文，确需联网时使用本地搜索工具，仍不足时再用其他联网手段

## 工程原则

| 原则 | 在本项目中的体现 |
|------|-----------------|
| **KISS** | 需求画像用 3-5 个问题而非冗长问卷；文档标题不加序号前缀 |
| **DRY** | 工作流定义在 AGENTS.md，系统提示词在 Prompts.md，各司其职避免重复 |
| **YAGNI** | 只实现当前推荐场景所需功能，不预设未来的多语言/多模态扩展 |

**原则冲突时**：正确性 > 简洁性 > 清晰性

## 变更管理

- 版本号由 `config.yaml` 的 `project_info.version` 统一管理
